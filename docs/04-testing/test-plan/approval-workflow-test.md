# 审批闭环功能测试说明

> 适用范围：审批意见、审批记录、审批通知、审批列表“最近审批人/时间”显示

## 1. 测试前置条件

1. 数据库已初始化（可删库重建）
2. 管理员账号可登录（默认 admin / admin123）
3. 至少存在一个普通用户账号（如 user01）
4. 至少存在一个会议室，且规则开启“需要审批”
   - 规则配置 `approvalRequired = true`

---

## 2. 功能测试用例

### 用例 A：普通用户提交预约 → 管理员收到审批通知

**步骤**
1. 普通用户登录
2. 选择会议室日历，创建预约（规则需审批）
3. 提交预约成功

**预期**
- 预约状态为 `PENDING`
- 管理员通知列表出现“有新的预约待审批”
- 管理员审批队列中能看到该预约

---

### 用例 B：管理员审批通过（带意见）

**步骤**
1. 管理员进入审批队列
2. 点击“通过”
3. 输入审批意见（可选）
4. 确认提交

**预期**
- 预约审批状态变为 `APPROVED`
- 操作日志新增一条 `APPROVE`
- 日志内容含“审批通过：{意见}”
- 普通用户通知中出现“预约已通过” + “审批意见：xxx”
- 审批队列列表“最近审批人/时间”字段更新

---

### 用例 C：管理员审批驳回（意见必填）

**步骤**
1. 管理员进入审批队列
2. 点击“驳回”
3. 不填写意见直接提交

**预期**
- 提示“驳回时请填写原因”
- 无法提交

继续：
4. 填写驳回意见
5. 提交

**预期**
- 预约审批状态变为 `REJECTED`
- 操作日志新增一条 `REJECT`
- 日志内容含“审批驳回：{原因}”
- 普通用户通知中出现“预约已拒绝” + “审批意见：xxx”

---

### 用例 D：审批记录展示

**步骤**
1. 管理员在审批队列点击“查看详情”
2. 普通用户在“我的预约”点击“查看详情”

**预期**
- 详情中展示审批记录
- 记录显示：操作类型、审批人、时间、意见内容

---

### 用例 E：审批人姓名记录

**步骤**
1. 确保管理员账号有 name
2. 完成一次审批（通过或驳回）
3. 查看审批记录

**预期**
- 日志中显示审批人姓名
- 若 name 为空，则显示 userId

---

## 3. API 测试点（可选）

### 3.1 获取审批详情
```
GET /api/bookings/{id}/detail
```
期望：
- 返回 booking + operationLogs
- operationLogs 含审批记录

### 3.2 审批通过
```
PUT /api/admin/bookings/{id}/approve
{
  "operatorId": "admin",
  "remark": "同意"
}
```
期望：
- 状态为 APPROVED
- 日志写入意见

### 3.3 审批驳回
```
PUT /api/admin/bookings/{id}/reject
{
  "operatorId": "admin",
  "remark": "时间冲突"
}
```
期望：
- 状态为 REJECTED
- 日志写入意见

## 4. 前端验证点

- 审批队列列表显示“最近审批人/时间”
- 审批对话框（通过/驳回）可输入意见
- 驳回意见必填
- 我的预约详情能看到审批记录
- 通知面板能看到审批意见摘要

---

## 5. 常见问题

### 5.1 管理员没有收到通知
- 检查是否开启审批
- 检查管理员是否登录
- 检查 NotificationBell 轮询是否启动

### 5.2 通知内容没有审批意见
- 检查审批时是否填写 remark
- 检查 booking_operation_log 是否写入

---
